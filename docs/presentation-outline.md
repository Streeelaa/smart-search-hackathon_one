# Presentation Outline

## Слайд 1. Проблема

Порталу поставщиков нужен поиск, который:

1. понимает опечатки и синонимы;
2. работает на языке каталога и на языке пользователя;
3. учитывает историю конкретного заказчика;
4. повышает качество верхушки выдачи.

## Слайд 2. Что мы сделали

1. Построили гибридный retrieval: keyword + semantic.
2. Добавили correction и synonym expansion.
3. Добавили rerank top-кандидатов.
4. Добавили динамический профиль пользователя.
5. Добавили demo UI, evaluation и импорт данных.

## Слайд 3. Архитектура

Поток:

запрос → нормализация → correction → synonyms → lexical retrieval + semantic retrieval → rerank → personalization → выдача → события → обновление профиля.

## Слайд 4. Персонализация

Профиль пользователя учитывает:

1. category affinity;
2. tag affinity;
3. recent queries;
4. average purchase price.

Как применяется:

1. не ломает релевантность;
2. переставляет релевантные результаты;
3. обновляется после событий click/favorite/purchase.

## Слайд 5. Объяснимость

Каждый результат содержит reasons:

1. совпадение в названии;
2. совпадение по тегу;
3. синоним;
4. semantic similarity;
5. personalization signal.

## Слайд 6. Метрики

Используем:

1. Hit@3
2. MRR@10
3. NDCG@10
4. Precision@3
5. Recall@10

Их роль:

1. оценка верхушки выдачи;
2. сравнение режимов поиска;
3. доказательство улучшений над baseline.

## Слайд 7. Демо

Показать 3 кейса:

1. опечатка;
2. синоним;
3. персонализация до и после.

## Слайд 8. Масштабирование

Что можно сделать после хакатона:

1. подключить реальные данные организаторов;
2. перейти на PostgreSQL в боевом режиме;
3. заменить retrieval на BGE-M3 или E5;
4. подключить более сильный reranker;
5. вынести индексацию в отдельный pipeline.

## Слайд 9. Почему решение сильное

1. не зависит от одного механизма поиска;
2. объяснимо;
3. расширяемо;
4. готово к реальным данным;
5. уже имеет UI, metrics и ingestion.
